# xboxlive-protect API — Rules

Block rules are the core data in xboxlive-protect.  The rules endpoints let you
list, create, edit, delete, and promote rules via the REST API.

All endpoints require a valid session cookie (see `docs/api-auth.md`) and
`must_change_password = false`.  Every authenticated request extends the session
by 30 days.

---

## Uniqueness model

A rule is uniquely identified by `(ip_address, cidr_prefix, source)`.  Two rows
for the same `(ip, cidr)` can coexist if they have different sources — for
example, a local rule and a subscription rule for `203.0.113.45/32` are two
distinct rows that both contribute to the blocklist.  This is a feature, not a
bug: when the subscription is removed, the local rule survives.

The nftables blocklist is a *projection* of all rules.  Overlapping CIDRs are
collapsed before being written to the kernel set, so a subscription /24 and a
local /32 inside it produce a single /24 element in nftables.

---

## IP validation

`POST /rules` rejects IPs that cannot safely be blocked:

| Reason | What triggers it |
|--------|------------------|
| `invalid` | Not a parseable IPv4 address or CIDR prefix |
| `loopback` | Overlaps 127.0.0.0/8 |
| `private` | Overlaps RFC 1918 (10/8, 172.16/12, 192.168/16) |
| `link-local` | Overlaps 169.254.0.0/16 |
| `xbox-live` | Overlaps the Xbox Live allowlist (inert until Phase 3 — see below) |
| `gateway` | Matches the device's detected default gateway IP |
| `bridge` | Matches the device's detected br0 IP |

**Xbox Live allowlist note (Stage 2):** The `xbox-live` check reads
`data/xbox-live-allowlist.json`, which is generated by Phase 3 from Microsoft's
Azure service tag JSON.  Until that file exists, the check logs a WARNING and
passes all IPs through.  A real Microsoft Xbox Live IP will return 201 in Stage 2
dev environments.  This will be corrected when Phase 3 is complete.

## Endpoints

### GET /api/v1/rules

List rules with optional filters and pagination.

**Query parameters:**

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `source` | `local` \| `subscription` \| `all` | `all` | Filter by rule origin |
| `since` | ISO 8601 datetime | — | Return rules where `created_at > since` |
| `search` | string | — | Case-insensitive substring match on `ip_address` or `comment` |
| `limit` | int 1–1000 | 100 | Page size |
| `offset` | int ≥ 0 | 0 | Page start |

**Response: 200**

```json
{
  "total": 42,
  "items": [
    {
      "id": 1,
      "ip_address": "203.0.113.45",
      "cidr_prefix": 32,
      "source": "local",
      "subscription_id": null,
      "comment": "spinbot host",
      "confidence": "high",
      "profile_id": "mw2-x360",
      "created_at": "2026-05-18T12:00:00",
      "updated_at": "2026-05-18T12:00:00"
    }
  ],
  "limit": 100,
  "offset": 0
}
```

`total` is the count before pagination, for building client-side pagers.

---

### POST /api/v1/rules

Add a local block rule.

**Request body:**

```json
{
  "ip_address": "203.0.113.45",
  "cidr_prefix": 32,
  "comment": "spinbot host",
  "confidence": "high"
}
```

| Field | Required | Notes |
|-------|----------|-------|
| `ip_address` | Yes | IPv4 address string |
| `cidr_prefix` | No | 0–32, default 32 |
| `comment` | No | Max 500 chars |
| `confidence` | No | `"high"` \| `"medium"` \| `"low"` |

**Responses:**

| Status | Meaning |
|--------|---------|
| 201 | Rule created; body is a `RuleResponse` |
| 409 | A local rule for this `(ip_address, cidr_prefix)` already exists |
| 422 | IP rejected; body is `{"error": "ip_not_blockable", "reason": "<reason>"}` |

A subscription rule for the same IP does not cause 409 — see the uniqueness
model above.

---

### PATCH /api/v1/rules/{id}

Edit a local rule.  Only `comment` and `confidence` are mutable.  `ip_address`
and `cidr_prefix` are the identity of the rule; to change them, delete and
recreate.

**Request body** (all fields optional; omitted fields are left unchanged):

```json
{
  "comment": "updated note",
  "confidence": "medium"
}
```

Setting a field to `null` explicitly clears it.  A PATCH whose values are
identical to the current row is a true no-op: 200 is returned immediately
without writing an audit entry.

**Responses:**

| Status | Meaning |
|--------|---------|
| 200 | Updated rule as `RuleResponse` |
| 403 | Rule is a subscription rule — use `POST /rules/{id}/promote` first |
| 404 | Rule not found |

---

### DELETE /api/v1/rules/{id}

Remove a local block rule and remove it from nftables.

**Responses:**

| Status | Meaning |
|--------|---------|
| 204 | Rule deleted |
| 403 | Rule is a subscription rule — use `POST /rules/{id}/promote` first |
| 404 | Rule not found |

---

### POST /api/v1/rules/{id}/promote

Promote a subscription rule to a local rule.  Sets `source = 'local'` and
clears `subscription_id`, detaching the rule from its subscription.  After
promotion the rule survives if the subscription is later deleted.

Idempotent: promoting an already-local rule returns 204 with no changes.

**Responses:**

| Status | Meaning |
|--------|---------|
| 204 | Promoted (or already local — no-op) |
| 404 | Rule not found |
| 409 | A local rule for the same `(ip_address, cidr_prefix)` already exists; promotion would create a duplicate |

## Audit log

Every mutating endpoint writes an `audit_log` entry:

| Endpoint | `event_type` | `target` |
|----------|-------------|---------|
| POST /rules | `rule_added` | `"ip/cidr"` |
| PATCH /rules/{id} (changed) | `rule_edited` | `"ip/cidr"` |
| DELETE /rules/{id} | `rule_removed` | `"ip/cidr"` |
| POST /rules/{id}/promote | `rule_edited` | `"ip/cidr"` |

All entries have a non-null `undo_token` (uuid4 hex string) reserved for the
Phase 4 undo stack.  The undo logic does not exist yet — the tokens are present
in the log so Phase 4 can implement against a populated history.

## RuleResponse shape

```json
{
  "id": 1,
  "ip_address": "203.0.113.45",
  "cidr_prefix": 32,
  "source": "local",
  "subscription_id": null,
  "comment": "optional note",
  "confidence": "high",
  "profile_id": "mw2-x360",
  "created_at": "2026-05-18T12:00:00",
  "updated_at": "2026-05-18T12:00:00"
}
```

`source` is `"local"` for user-created rules or `"subscription:<id>"` for rules
synced from a subscription.  `confidence` is `"high"`, `"medium"`, `"low"`, or
`null`.

---

## Error shapes

**IP not blockable (422):**
```json
{"detail": {"error": "ip_not_blockable", "reason": "private"}}
```

**Subscription rule immutable (403):**
```json
{
  "detail": {
    "error": "subscription_rule_immutable",
    "message": "Subscription rules are read-only. Use POST /rules/{id}/promote to convert to a local rule first."
  }
}
```

**Rule already exists (409):**
```json
{
  "detail": {
    "error": "rule_already_exists",
    "message": "A local rule for this (ip_address, cidr_prefix) already exists."
  }
}
```

**Forced password change (403) — applies to all endpoints:**
```json
{"detail": {"error": "password_change_required", "message": "..."}}
```

## curl examples

```bash
BASE=https://xboxlive-protect.local
COOKIE="xblp_session=<your-session-token>"

# List all rules
curl -sk -H "Cookie: $COOKIE" "$BASE/api/v1/rules"

# Add a rule
curl -sk -X POST -H "Cookie: $COOKIE" -H "Content-Type: application/json" \
  -d '{"ip_address":"203.0.113.45","comment":"test"}' \
  "$BASE/api/v1/rules"

# Edit a rule
curl -sk -X PATCH -H "Cookie: $COOKIE" -H "Content-Type: application/json" \
  -d '{"comment":"updated"}' \
  "$BASE/api/v1/rules/1"

# Delete a rule
curl -sk -X DELETE -H "Cookie: $COOKIE" "$BASE/api/v1/rules/1"

# Filter by source
curl -sk -H "Cookie: $COOKIE" "$BASE/api/v1/rules?source=local"

# Search
curl -sk -H "Cookie: $COOKIE" "$BASE/api/v1/rules?search=203.0.113"

# Paginate
curl -sk -H "Cookie: $COOKIE" "$BASE/api/v1/rules?limit=10&offset=20"

# Promote a subscription rule
curl -sk -X POST -H "Cookie: $COOKIE" "$BASE/api/v1/rules/5/promote"
```
